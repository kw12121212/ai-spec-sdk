# Design: rate-limiting

## Approach

在 HTTP 中间件层实现令牌桶限流，位置在认证/授权检查之后、方法分发之前。

**令牌桶实现：**
- 用 `Map<keyId, { tokens: number, lastRefill: number }>` 在内存中维护每个 Key 的桶状态
- 桶容量 = 速率限制 = 120（突发上限与持续上限相同，简单实现）
- 每次请求时按时间差补充令牌：`tokensToAdd = elapsed_ms / 60000 * 120`，上限不超过桶容量
- 若令牌 ≥ 1，消耗 1 个令牌，放行请求；否则返回 429

**中间件位置：**
```
请求 → 认证 → 授权 → 限流检查 → 方法分发
```
认证失败（`-32061`）和授权失败（`-32060`）在限流之前短路，保持现有错误优先级。

**429 响应格式：**
```json
{
  "jsonrpc": "2.0",
  "id": <request_id>,
  "error": {
    "code": -32029,
    "message": "Rate limit exceeded"
  }
}
```
HTTP 状态码 429，附带响应头：
- `X-RateLimit-Limit: 120`
- `X-RateLimit-Remaining: 0`
- `X-RateLimit-Reset: <unix_timestamp_seconds>` — 下一个令牌补充到 1 的时间点

## Key Decisions

- **错误码 `-32029`**：JSON-RPC 规范保留 `-32000` 至 `-32099` 为服务器错误，`-32029` 与现有 `-32061`（未认证）、`-32060`（未授权）、`-32050`（版本协商）保持同一命名空间风格。
- **桶容量 = 速率上限**（无额外突发）：避免引入两个参数，保持实现简单；未来可扩展。
- **进程内存存储**：无外部依赖，重启后重置符合开发场景预期，生产部署通常单实例。
- **admin 绕过在令牌桶检查前判断**：避免 admin Key 消耗桶槽，确保管理操作永不被限流。
- **`--no-auth` 时不启用限流**：无 Key 标识时无法按 Key 限流，跳过整个中间件更一致。

## Alternatives Considered

- **滑动窗口计数器**：精度更高，但实现复杂、内存占用略高，对本场景收益有限。
- **固定窗口计数器**：实现最简单，但窗口边界处允许 2× 突发流量，令牌桶更平滑。
- **按 Key+端点粒度**：用户明确要求仅按 Key，避免过早复杂化。
- **外部存储（Redis）**：多实例场景有价值，但当前 Bridge 为单进程模型，引入依赖不值得。
